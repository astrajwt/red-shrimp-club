# 多 Agent / Agent Memory / Memory Task 设计简述

> 目的：用一份短文档把这个项目最核心的产品思路讲清楚。  
> 范围：只讨论三件事
> 1. 多 agent 协作
> 2. agent memory 设计
> 3. memory task 设计

---

## 1. 设计目标

这个项目不是“做一个能聊天的 AI 列表”，而是做一个可以长期运转的多 agent 工作系统。

核心目标有 3 个：

1. 用户面对的是一个团队，而不是单个模型调用入口。
2. 每个 agent 都有持续积累的 memory，而不是一次性上下文。
3. 所有工作最好都能沉淀成 task 和文档，而不是只存在于聊天记录里。

换句话说，这个系统想解决的问题不是“怎么让模型答一句话”，而是：

- 怎么让多个 agent 分工
- 怎么让 agent 记住项目
- 怎么让执行过程留下可复查、可复用的工作材料

---

## 2. 多 Agent 设计

### 2.1 为什么要多 agent

单 agent 容易遇到 3 个问题：

1. 角色混杂：一个 agent 同时负责调度、研发、研究、汇报，容易失焦。
2. 上下文污染：所有任务都塞进一个 memory，后续越来越重。
3. 并行度不足：用户的多个请求不能自然拆开并发推进。

因此系统需要把 agent 设计成“长期存在的团队成员”，每个 agent 有自己的：

- 名字
- 角色
- runtime / model
- workspace
- memory
- task 责任范围

### 2.2 多 agent 的最小原则

多 agent 不是越多越好，而是要满足下面几个原则：

1. 有明确角色边界。
2. 能被 task 明确指派。
3. 有独立 workspace 和 memory。
4. 可以被上级 agent 或人类协调。

### 2.3 层级关系

系统里的 agent 不应该只是平铺列表，而应该允许形成“汇报关系”。

原因：

1. 复杂任务需要有人负责拆解和汇总。
2. 用户不想逐个盯所有 agent。
3. 某些 agent 更像执行者，某些 agent 更像协调者。

所以 agent 需要支持：

- `reports to`
- 默认上级
- 层级展示

这套层级不是为了做组织架构图，而是为了把“谁负责收口”明确下来。

## 3. Agent Memory 设计

### 3.1 Memory 不是聊天记录

agent memory 的核心思想是：

- 聊天记录是过程噪音
- memory 是提炼后的长期工作知识

因此 memory 不应该只是自动把消息堆起来，而应该变成 agent 的工作底稿和长期记忆。

### 3.2 Memory 的三层结构

当前设计更适合拆成三层：

1. `MEMORY.md`
   - 记录 agent 的身份、当前上下文、重要偏好、最近工作状态
   - 作用是“让 agent 下次启动时迅速进入状态”

2. `KNOWLEDGE.md`
   - 记录较稳定的领域知识、架构约束、长期规则
   - 作用是“沉淀不会因为单个任务结束就消失的知识”

3. `notes/`
   - 记录任务过程中的中间文档、阅读笔记、分析稿、方案稿
   - 作用是“保留工作材料”

这三层的区别是：

- `MEMORY.md` 偏运行时状态
- `KNOWLEDGE.md` 偏长期知识
- `notes/` 偏任务产物

### 3.3 Memory 系统需要支持的能力

为了让 memory 真正有用，而不是一堆 markdown 文件，系统至少要支持：

1. 文件树浏览
2. 文档查看
3. `[[wikilink]]` 跳转
4. backlinks / outgoing links
5. git 导入外部知识库
6. 复制真实可用的文档路径

这里有一个关键思想：

memory 不只是“展示 markdown”，而是要让这些文档成为 agent 工作流里的可操作对象。

比如：

- 文档可以被挂到 task
- 文档可以被其他文档引用
- 文档可以成为 agent 回答问题时的上下文来源

### 3.4 为什么 memory 要放在文件系统里

因为文件系统 + markdown 的好处非常直接：

1. 可读
2. 可 git 管理
3. 可人工修改
4. 可被 Obsidian 一类工具直接使用
5. 可被 agent 直接产出和复用

这比把所有 memory 塞进数据库字段更灵活，也更接近真实知识工作的形态。

## 4. Memory Task 设计

### 4.1 Task 不只是状态卡片

普通任务系统常常只管：

- 标题
- 状态
- 指派

但在这个项目里，这还不够。

因为 agent 的真正工作过程需要文档化，所以 task 必须和 memory 联动。

### 4.2 Memory Task 的核心思想

每个 task 最好都能挂上与它有关的文档。

这些文档可能来自：

1. 人工补挂现有 memory 文档
2. 创建 task 时自动生成根文档
3. agent 执行期间自动写出的 note

这样 task 就不再只是一个标题，而是一个“带工作材料的任务包”。

### 4.3 为什么 task 要和文档绑定

绑定的价值有 4 个：

1. 审查时有依据
2. 交接时有材料
3. memory 不会散落
4. 后续可提炼为 skill / knowledge

这也是为什么系统里需要：

- `task_documents`
- 文档已读状态
- `link doc`
- agent 自动追加 todo note

### 4.4 推荐的 task 生命周期

一个更合理的 memory-task 流程应该是：

1. 创建 task
2. 明确 assignee
3. 自动或手工挂上根文档
4. agent 在执行中持续追加 note / plan / review material
5. task 进入 review 时，reviewer 不只看状态，还看 linked docs
6. task 完成后，相关文档继续留在 memory 中沉淀

这里最关键的一点是：

task 是执行入口，memory 是执行痕迹。

两者必须联动，系统才能真正形成“可复盘的 agent 工作流”。

---

## 5. 三套设计之间的关系

这三部分其实是一个闭环：

1. 多 agent 负责分工
2. task 负责组织工作
3. memory 负责沉淀工作结果

更具体地说：

- agent 决定“谁做”
- task 决定“做什么”
- memory 决定“做过什么，学到了什么”

如果只有 agent 没有 task，系统会变成多人闲聊。
如果只有 task 没有 memory，系统会变成没有材料的工单板。
如果只有 memory 没有 agent，系统又只是一个知识库。

这个项目的价值，正是在于把三者接起来。

---

## 6. 设计边界

这份文档只主张下面这些最核心的设计，不扩展到更重的企业能力：

- 不先讨论复杂权限系统
- 不先讨论计费系统
- 不先讨论复杂项目管理模型
- 不先讨论完整组织树和审批流

先把下面三件事做好更重要：

1. agent 可长期运行和协作
2. memory 可被持续积累和调用
3. task 可和 memory 强绑定

---

## 7. 一句话总结

这个项目的核心思想可以压缩成一句话：

把 AI 从“会话回答器”变成“有角色、有记忆、有任务材料沉淀的长期团队成员”。

多 agent 解决分工，agent memory 解决长期记忆，memory task 解决执行过程可复盘。
