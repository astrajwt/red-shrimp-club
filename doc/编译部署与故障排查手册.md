# 编译部署与故障排查手册

> 适用仓库：`/home/jwt/workspace/slock-clone`
> 当前默认服务端口：`3001`

## 1. 日常编译与重启

### 1.1 重编前端

```bash
cd /home/jwt/workspace/slock-clone/frontend-src
npm run build
```

已知旧 warning：
- PostCSS `@import` 顺序 warning
- Vite chunk size warning

这两个目前不是阻塞问题。只要 build 退出码是 `0`，前端产物就算成功。

### 1.2 重编后端

```bash
cd /home/jwt/workspace/slock-clone/backend-src
npm run build
```

### 1.3 重启 3001 后端

先停旧进程：

```bash
pkill -f 'node dist/index.js'
```

再启动新进程：

```bash
cd /home/jwt/workspace/slock-clone/backend-src
nohup npm run start >/tmp/redshrimp-backend.log 2>&1 &
```

如果你想直接看启动日志，也可以前台启动：

```bash
cd /home/jwt/workspace/slock-clone/backend-src
npm run start
```

看到下面这种输出，说明监听已经起来：

```text
Server listening at http://127.0.0.1:3001
Backend running at http://0.0.0.0:3001
```

## 2. 重启后必查

### 2.1 看进程

```bash
pgrep -af 'node dist/index.js'
```

### 2.2 看健康检查

```bash
curl -sf http://127.0.0.1:3001/health
curl -sf http://192.168.1.2:3001/health
```

期望结果：

```json
{"status":"ok","ts":"..."}
```

### 2.3 看最新日志

```bash
tail -n 50 /tmp/redshrimp-backend.log
```

## 3. 首页 / Dashboard 黑屏排查

这类问题先别凭感觉判断“服务挂了”。按下面顺序查。

### 3.1 先判断是不是后端没起来

```bash
curl -sf http://127.0.0.1:3001/health
```

如果失败：
- 先重启 backend
- 再看 `/tmp/redshrimp-backend.log`

### 3.2 再判断 dashboard 路由是否存在

```bash
curl -i http://127.0.0.1:3001/api/bulletins/dashboard
```

结果含义：
- `404`：后端还是旧进程，新路由没加载，先重启 backend
- `401`：路由已经加载，只是当前请求没带登录态，这说明不是“后端没更新”
- `200`：后端路由正常，继续查前端数据渲染

这次主页纯黑，真实根因就是：
- backend 是旧进程
- `/api/bulletins/dashboard` 返回 `404`
- 前端首页新逻辑没拿到数据，结果黑屏

### 3.3 如果路由正常，再看是不是前端运行时错误

优先检查：
- 浏览器 console 第一条报错
- `Quick Links / Sticky Notes / Leader Lanes` 是否有 `null` 数据直接 `map`
- 首页请求失败时，前端有没有正确展示 error state，而不是继续渲染空数据

## 4. 访问异常怎么区分

### 4.1 `ERR_CONNECTION_REFUSED`

通常表示：
- backend 根本没起来
- 或端口没监听

先查：

```bash
pgrep -af 'node dist/index.js'
ss -ltnp | grep 3001
```

### 4.2 `ERR_EMPTY_RESPONSE`

通常表示：
- 进程刚启动就退出
- 或中途 crash

先查：

```bash
tail -n 100 /tmp/redshrimp-backend.log
```

如果 `nohup` 启动后进程马上消失，就直接前台跑一次：

```bash
cd /home/jwt/workspace/slock-clone/backend-src
npm run start
```

用前台输出拿真实错误。

### 4.3 本机可用，局域网不可用

先查 `HOST`：

```bash
grep '^HOST=' /home/jwt/workspace/slock-clone/backend-src/.env
ss -ltnp | grep 3001
```

如果绑定成 `127.0.0.1:3001`，局域网设备打不开。要改成：

```bash
HOST=0.0.0.0
```

然后重启 backend。

## 5. 发布前最小检查清单

每次准备上线，至少过一遍：

```bash
cd /home/jwt/workspace/slock-clone/frontend-src && npm run build
cd /home/jwt/workspace/slock-clone/backend-src && npm run build
pkill -f 'node dist/index.js'
cd /home/jwt/workspace/slock-clone/backend-src && nohup npm run start >/tmp/redshrimp-backend.log 2>&1 &
curl -sf http://127.0.0.1:3001/health
curl -sf http://192.168.1.2:3001/health
curl -i http://127.0.0.1:3001/api/bulletins/dashboard
```

发布前看到下面这组状态，才算基本可交付：
- 前后端 build 都是 `0`
- `health` 本机和局域网都通
- 新增 API 路由不再是 `404`
- 页面刷新后不是黑屏

## 6. 结论

这个项目现在最常见的线上问题，不是“代码没写对”，而是：
- 改了前后端但没重启 backend
- 前端吃到了新 bundle，后端还是旧路由
- 局域网监听地址写错

所以碰到问题时，先查：
1. build 过没过
2. backend 是不是新进程
3. `/health` 是否正常
4. 目标 API 是 `404`、`401` 还是 `200`

这四步比盲猜快很多。
